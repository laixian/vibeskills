# 产品洞察生成器

对当前项目进行产品层面的深度分析，不评判代码质量，不做泛泛的批评，而是**找出产品愿景与当前实现之间的真正差距**，输出有针对性的洞察。

**核心原则：先理解产品的灵魂，再用创始人的愿景去照出错位。没有理解就没有洞察。**

---

## 第一步：找到产品的灵魂

目标：理解产品为用户编排的**核心工作流**，而不是代码架构。

### 1.1 识别产品的工作流

产品的核心工作流不一定被显式定义在某个文件里，但一定会在代码中留下痕迹。你需要从以下**四类信号**中拼出完整的工作流，而不是只去某个固定目录找文件。

#### 信号 A：状态与生命周期

搜索项目中的**状态枚举、状态机、生命周期定义**。这是最强的工作流信号——状态流转就是工作流的骨架。

寻找线索：
- 枚举定义中带有 `status`、`state`、`stage`、`phase` 等字段
- 数据模型中的状态字段（`status: todo → doing → done`）
- 前端代码中的步骤组件（Stepper、Wizard、多阶段表单）
- 状态变更的 API 端点（`PATCH /status`、`/approve`、`/submit`）

找到状态定义后问自己：**谁触发了状态变更？是用户手动操作，还是系统自动推进？**这个问题的答案直接揭示了产品的自动化程度。

#### 信号 B：用户入口与旅程

从用户能触达的**入口**出发，追踪他们的操作路径。

寻找线索：
- **前端路由结构**：页面路径反映用户可见的功能面（`/dashboard`、`/projects/{id}/settings`）
- **CLI 命令树**：子命令结构就是用户的操作菜单（`app deploy`、`app logs`、`app rollback`）
- **导航组件 / 侧边栏**：展示了产品认为最重要的功能入口
- **Onboarding 流程**：新用户引导揭示了产品认为的"核心路径"

#### 信号 C：自动化与编排

搜索产品中**系统替用户做的事**，这是产品设计意图的直接体现。

寻找线索：
- **后台任务 / 队列**：Celery tasks、Bull jobs、cron jobs、BackgroundTasks
- **Webhook / 事件监听**：`on_create`、`on_status_change`、event handlers
- **触发器与钩子**：Git hooks、数据库 triggers、lifecycle hooks
- **编排脚本**：Skill 文件、Makefile、CI/CD pipeline 定义
- **中间件链**：请求处理管道中自动执行的逻辑（auth → validate → transform → respond）

这些自动化的存在说明产品设计者认为这些操作**不应该由用户手动完成**——这本身就是一种产品判断。

#### 信号 D：集成与数据流向

搜索产品与**外部系统的连接点**，这揭示了产品在更大工作流中的位置。

寻找线索：
- **第三方 SDK / API 客户端**：`import slack_sdk`、`new Octokit()`、飞书/钉钉集成
- **OAuth 配置**：产品连接了哪些外部身份源
- **Import / Export 功能**：数据从哪里来、到哪里去
- **Webhook 发送**：产品主动通知哪些外部系统

集成的方向很重要：**只读**（从外部拉数据）、**只写**（向外部推数据）、还是**双向**（读写都有）？双向集成意味着产品试图成为工作流的中枢，而非末端。

---

**综合四类信号后，你应该能回答：** 这个产品把用户从什么状态带到什么状态？中间哪些步骤是自动的？数据流经了哪些系统？

### 1.2 提取核心循环

读完上述文件后，在脑中画出产品的**核心使用循环**：

- 用户从哪里开始？（入口）
- 经过哪些步骤达成目标？（路径）
- 系统在每一步自动做了什么？（编排）
- 产出物回流到哪里？（闭环）

**检验标准：能否用一张流程图画出产品的核心循环？如果画不出来，说明还没理解透，继续读。**

### 1.3 理解数据流

快速扫描数据模型（ORM 模型 / Schema 定义 / 数据库迁移），关注：

- 核心实体之间的关系
- 数据的生产者和消费者
- 是否存在数据回流（A 的产出成为 B 的输入，B 的产出又反哺 A）

数据回流的存在意味着产品有**飞轮效应**的潜力。

---

## 第二步：向用户提出一个精准问题

在完成第一步的理解后，**必须向用户提问**。不要跳过这一步。

提问模板：

```
我已经理解了你的产品核心流程：

{用 3-5 句话概括你理解的核心循环}

在我给出产品洞察之前，想确认一个关键问题：

**你希望用户因为什么原因离不开这个产品？**
换个说法：如果这个产品明天消失了，用户会因为失去什么而难受？

（这不是在问功能列表，而是在问产品的核心价值主张。）
```

**为什么这个问题不可省略：** 代码只能告诉你"做了什么"，不能告诉你"想做什么"。产品愿景必须从创始人口中获得，任何试图从代码推导愿景的尝试都会导致泛泛而谈的伪洞察。

等待用户回答后再继续。

---

## 第三步：用愿景照当前形态，找错位

拿用户的回答（愿景），逐一对照第一步画出的核心循环，从四个维度寻找错位：

### 3.1 体验矛盾

循环中哪个环节的**用户体验**和愿景矛盾？

- 愿景说"低门槛"，但入口需要 5 步操作？
- 愿景说"快速验证"，但从想法到看到结果要等 2 天？
- 愿景说"团队协作"，但关键操作只能在开发者终端完成？

### 3.2 断裂的环节

循环中哪个环节**断了**，导致闭环无法形成？

- 有输入没输出？（用户提交了东西但没有得到反馈）
- 有产出没有回流？（做完了但知识没有沉淀回系统）
- 有起点没有终点？（发起了流程但没有明确的"完成"标志）

### 3.3 管道瓶颈

循环中哪个环节是**最慢的**，拖慢了整条管道？

- 是否有人工审批阻塞了自动化流程？
- 是否有同步等待可以改为异步？
- 是否有手动操作可以自动化？

### 3.4 未被充分利用的资产

产品中是否有已经积累但**没有被充分利用**的数据或能力？

- 沉淀的历史数据是否只被被动查阅，而没有主动产生价值？
- 已有的集成能力是否只用了一个方向？（只写不读，或只读不写）
- 用户的行为数据是否被捕获并反馈到产品决策中？

---

## 第四步：识别护城河

回答一个问题：**如果竞品明天抄了所有功能，用户为什么还留在这里？**

通常答案是以下之一：

- **数据积累**：用得越久，系统越懂用户/项目，迁移成本越高
- **网络效应**：用的人越多，每个人的价值越大
- **工作流锁定**：产品嵌入了用户的日常流程，替换成本高
- **独特能力**：有某个核心能力是竞品无法快速复制的

识别出护城河后，评估：当前的产品设计是否在**主动加深**这条护城河？还是在护城河之外的地方分散精力？

---

## 第五步：生成洞察文档

将分析结果写入 `docs/product-insights.md`（如已存在则覆盖）。

文档结构：

```markdown
# {项目名称} — 产品洞察

> 生成时间：{通过 `date '+%Y-%m-%d %H:%M'` 获取的本地时间}

## 产品核心循环

用流程图或文字描述产品的核心使用循环。这是后续所有分析的基础。

## 创始人愿景

记录用户对"产品不可替代性"的回答（原话）。这是衡量所有洞察的标尺。

## 洞察

每个洞察独立成节，结构为：

### 洞察 N：{一句话标题}

**现状**：当前产品在这个点上是怎样的（基于事实，引用具体的功能/流程）

**错位**：这与创始人愿景之间的矛盾是什么（具体、尖锐、不回避）

**建议方向**：如果要消除这个错位，可以怎么做（给方向，不给实现细节）

---

（重复上述结构，通常 3-6 个洞察）

## 护城河分析

当前的护城河是什么、有多深、是否在被主动加深。

## 行动优先级

| 优先级 | 方向 | 预期效果 |
|--------|------|----------|
| P0 | ... | ... |
| P1 | ... | ... |
| P2 | ... | ... |

优先级判断标准：
- P0：直接阻碍核心愿景的实现
- P1：显著加速飞轮或加深护城河
- P2：锦上添花，提升体验但不改变格局
```

### 写作原则

- **每个洞察必须同时包含"现状事实"和"愿景对比"**，缺一个都不算洞察
- **不做泛泛的批评**（"没有测试""缺少文档"这类对任何项目都成立的话不要写）
- **不做功能建议**（"你应该加一个 XX 功能"不是洞察，"你的入口太重导致 idea 流失"才是）
- **要尖锐但有建设性**——指出问题的同时给出方向
- 用中文撰写

---

## 反模式清单（不要做的事）

以下是生成伪洞察的常见陷阱，必须避免：

1. **没识别工作流信号就开始分析** → 会产出对任何产品都成立的废话（四类信号：状态生命周期、用户入口旅程、自动化编排、集成数据流向）
2. **没问创始人愿景就评价产品** → 会用自己假设的标准去衡量，导致"为了拷问而拷问"
3. **从代码质量角度评价产品** → 代码质量是工程问题，不是产品洞察
4. **列举缺失的功能** → 功能清单不是洞察，错位分析才是
5. **给出技术实现方案** → 这份文档的读者是产品决策者，不是开发者
